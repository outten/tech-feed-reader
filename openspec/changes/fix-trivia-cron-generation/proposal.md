## Why

The daily News Trivia quiz reliably generates when run manually (`make generate-trivia` / `scripts/generate_trivia.rb`), but the identical logic invoked via the `generate_trivia` sidekiq-cron entry (`GenerateTriviaWorker`, 01:30 UTC) does not reliably produce a quiz, in both the Droplet and local dev. Local `tmp/logs/sidekiq.log` history shows the concrete failure mode: several cron-triggered runs raised `NameError: uninitialized constant GenerateTriviaWorker`, retried twice, and landed in the Sidekiq dead set — while a manually-run `scripts/generate_trivia.rb` process (which requires its own dependencies directly and never touches the Sidekiq class-resolution path) has no such failure mode. Users get "no quiz today" with nothing in the app's own structured logs to explain why, since the failure happens before `GenerateTriviaWorker#perform` — and therefore `AppLogger` — ever runs.

## What Changes

- Confirm the failure signature on the Droplet itself (`docker compose logs sidekiq`, `/admin/sidekiq` dead set, `/admin/status` cron panel) before changing code, since the only hard evidence so far is from local dev logs.
- Harden local dev process lifecycle (`scripts/run_all.sh`, `scripts/stop_all.sh`, `make sidekiq`) so a stale/duplicate Sidekiq worker process can never linger and silently steal cron-enqueued jobs out of the shared Redis queue — the same `pkill`-before-start pattern `make stop` already uses for the web process, applied to the Sidekiq process.
- Add a distinct, pre-`perform` observability signal for sidekiq-cron jobs that fail before the app's own logger runs (i.e., surface Sidekiq-level dead jobs / class-resolution failures somewhere more visible than raw container logs), so a future recurrence is caught the same day rather than discovered a month later.
- Verify the existing dead-job ntfy alert (`app/sidekiq_config.rb` death_handler) is actually configured (`NTFY_URL` set) on the Droplet — if it is, a past dead `GenerateTriviaWorker` job should already have paged; if it isn't, that's a separate gap worth closing regardless of this bug's root cause.
- Once root cause is confirmed, apply the targeted fix (process-hygiene fix in dev tooling, and/or Droplet-side fix if the confirmed cause differs, e.g. a leftover non-Docker sidekiq process from before the Docker migration).

## Capabilities

### New Capabilities
- `trivia-generation-reliability`: The daily News Trivia quiz must be generated by the scheduled sidekiq-cron job with the same reliability as manual invocation, and any failure to do so must be observable (logged/alerted) same-day rather than silently discovered later.

### Modified Capabilities
(none — no existing spec covers trivia generation or sidekiq-cron job reliability)

## Impact

- `scripts/run_all.sh`, `scripts/stop_all.sh` — dev process lifecycle hygiene for the Sidekiq worker.
- `app/workers/generate_trivia_worker.rb`, `config/sidekiq_cron.yml` — only if the confirmed root cause requires a code/config change here (e.g. deploy-side stale process).
- `docs/alerting.md` — clarify/confirm `NTFY_URL` configuration expectations for dead-job alerts.
- No user-facing behavior changes and no database/API changes expected; this is an operational reliability fix for an existing background job.
