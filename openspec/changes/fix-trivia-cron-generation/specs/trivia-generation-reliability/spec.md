## ADDED Requirements

### Requirement: Scheduled trivia generation matches manual generation reliability
The daily News Trivia quiz SHALL be generated by the `generate_trivia` sidekiq-cron job (`GenerateTriviaWorker`) with the same reliability as running `scripts/generate_trivia.rb` directly. A cron-triggered run SHALL NOT fail due to the job's class or its dependencies being unresolvable in the worker process that picks it up.

#### Scenario: Cron-triggered run succeeds like the manual script
- **WHEN** the `generate_trivia` cron job fires at 01:30 UTC and no quiz exists yet for today
- **THEN** `GenerateTriviaWorker#perform` runs to completion and a quiz row exists for today, the same outcome as running `make generate-trivia` manually

#### Scenario: Only one worker process serves the job queue in dev
- **WHEN** a developer runs `make run-all` (or restarts it) in local dev
- **THEN** at most one Sidekiq worker process is alive and connected to Redis at any time, so a cron-enqueued job can never be picked up by a stale process that lacks the current code loaded

### Requirement: Pre-perform job failures are observable same-day
A Sidekiq job failure that occurs before the job's own application logging runs (e.g. a class-resolution failure) SHALL still be observable to the operator on the day it happens, not only discoverable later by reading raw container logs.

#### Scenario: A dead job pages the operator
- **WHEN** any Sidekiq job (including `GenerateTriviaWorker`) exhausts its retries and lands in the dead set
- **THEN** the configured dead-job alert (`Notifier.push` via `death_handlers` in `app/sidekiq_config.rb`) fires, provided `NTFY_URL` is configured on the Droplet

#### Scenario: Alert configuration is confirmed, not assumed
- **WHEN** this change is implemented
- **THEN** the operator has explicitly confirmed whether `NTFY_URL` is set on the Droplet, so silence is known to mean "no failures" rather than "alerting isn't wired up"
